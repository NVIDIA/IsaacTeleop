Teleop Control Protocol#

Two-way control channel between a PC operator dashboard and an XR headset running the Isaac Teleop web client (deps/cloudxr/webxr_client). The hub runs inside the existing WSS proxy (wss.py) when started with --enable-hub.

Enabling the Hub#

python -m isaacteleop.cloudxr --enable-hub

All routes below become active on the same TLS port as the CloudXR WSS proxy (default 48322). Without --enable-hub the proxy behaves exactly as before; teleop paths return 404.

Optional environment variable overrides for containers/systemd:

TELEOP_CONTROL_HUB=1    # equivalent to --enable-hub
CONTROL_TOKEN=<secret>  # require token on register / dashboard commands

To run only the WSS proxy and hub (no CloudXR runtime), use --wss-only together with --enable-hub — see Local testing (hub + WSS without starting the runtime).

WebSocket Endpoint#

URL

wss://<host>:<PROXY_PORT>/teleop/v1/ws

Subprotocol

none

Wire format

Text frames, UTF-8 JSON, one object per message

Message Envelope#

Every message in both directions uses the same envelope:

{ "type": "<string>", "payload": { } }

An unknown type causes the server to send an error message and ignore the payload.

Messages: Client → Server#

register#

First message after connecting. Establishes role and validates the token (if required).

Headset:

{
  "type": "register",
  "payload": {
    "role": "headset",
    "token": "<optional; required when CONTROL_TOKEN set>",
    "deviceLabel": "<optional human-readable name>"
  }
}

Dashboard (operator):

{
  "type": "register",
  "payload": {
    "role": "dashboard",
    "token": "<optional; required when CONTROL_TOKEN set>"
  }
}

The server responds with a hello message. If the token is invalid the server sends an error (code UNAUTHORIZED) and closes the connection.

setConfig (dashboard only)#

Updates the hub’s authoritative StreamConfig, increments configVersion, and pushes a config message to the connected headset(s).

{
  "type": "setConfig",
  "payload": {
    "token": "<optional>",
    "targetClientId": "<optional uuid; omit to push to all headsets>",
    "config": {
      "serverIP": "192.168.1.100",
      "port": 48322
    }
  }
}

payload.config may be partial; the hub shallow-merges it over the current StreamConfig. configVersion is always monotonically incremented by the hub.

command (dashboard only)#

Sends a connect or disconnect command to a headset.

{
  "type": "command",
  "payload": {
    "token": "<optional>",
    "targetClientId": "<optional uuid; omit if exactly one headset connected>",
    "action": "connect"
  }
}

action: connect | disconnect.

If targetClientId is omitted and multiple headsets are connected, the hub returns an error (code CONFLICT). Otherwise the command is forwarded to the headset as a sessionCommand.

clientMetrics (headset only)#

The headset reports latest performance metrics. The hub stores the most-recent values per cadence.

{
  "type": "clientMetrics",
  "payload": {
    "t": 1730000000000,
    "cadence": "frame",
    "metrics": {
      "streaming.framerate": 72.0,
      "render.framerate": 90.0,
      "render.pose_to_render_time": 12.5
    }
  }
}

cadence: "frame" | "render" (matches CloudXR MetricsCadence). metrics: string key → number value. Use the wire strings from CloudXR MetricsName (for example streaming.framerate, render.framerate, render.pose_to_render_time), not the TypeScript enum identifiers.

Messages: Server → Client#

hello#

Sent immediately after a successful register.

To headset:

{
  "type": "hello",
  "payload": {
    "clientId": "uuid",
    "configVersion": 1,
    "config": { "<StreamConfig>" }
  }
}

To dashboard:

{
  "type": "hello",
  "payload": {
    "clientId": "uuid"
  }
}

config (headset only)#

Pushed whenever the hub’s StreamConfig changes (after setConfig).

{
  "type": "config",
  "payload": {
    "configVersion": 2,
    "config": { "<StreamConfig — full merged object>" }
  }
}

Note

Headset behavior when config arrives during an active XR session: apply the new settings to in-memory state immediately; the next connection (after a disconnect) uses the new target. Do not tear down the existing XR session.

sessionCommand (headset only)#

Forwarded from a dashboard command.

{
  "type": "sessionCommand",
  "payload": {
    "action": "connect",
    "requestId": "<optional opaque string>"
  }
}

action: connect | disconnect.

error (any client)#

{
  "type": "error",
  "payload": {
    "code": "UNAUTHORIZED | BAD_REQUEST | NOT_FOUND | CONFLICT",
    "message": "<human readable>",
    "requestId": "<optional; echo from request>"
  }
}

The hub stores one merged StreamConfig plus a configVersion counter. setConfig shallow-merges (top-level keys only). The headset applies the hub config to its CloudXR connection settings before executing sessionCommand.connect.

HTTP Snapshot#

GET /api/teleop/v1/state returns the following JSON:

{
  "updatedAt": 1730000000000,
  "configVersion": 2,
  "config": { "<StreamConfig>" },
  "headsets": [
    {
      "clientId": "uuid",
      "connected": true,
      "deviceLabel": "string or null",
      "registeredAt": 1730000000000,
      "metricsByCadence": {
        "frame": { "at": 1730000000000, "metrics": { "streaming.framerate": 72.0 } },
        "render": { "at": 1730000000000, "metrics": { "render.framerate": 90.0 } }
      }
    }
  ],
  "dashboards": [
    {
      "clientId": "uuid",
      "connected": true,
      "registeredAt": 1730000000000
    }
  ]
}

Note

No logs or recentLogs fields in v1.

Headset Query Parameters#

The WebXR client reads these from the page URL when the app loads (standard query string on the same HTTPS origin that serves webxr_client). They are not entered in a separate form inside the app unless you add one; the usual approach is to open a single long URL in the headset browser.

Parameter

Description

controlWsUrl

Full WSS URL for the hub, e.g. wss://pc.example:48322/teleop/v1/ws. If absent, the control channel is disabled.

controlToken

Token sent in the register message. Must match the CONTROL_TOKEN environment variable if set on the server.

Getting controlWsUrl onto the headset (avoid typing)#

Typing a long wss://… URL in a XR keyboard is error-prone. Prefer copy–paste or a link opened directly on the device.

1. Build the correct hub address on the PC

  • Open the operator dashboard in a desktop browser: https://<pc-host>:<PROXY_PORT>/teleop/ (see Local testing (hub + WSS without starting the runtime) for local TLS notes).

  • If the headset is not on the same machine as the hub, open the dashboard using the PC’s LAN IP or DNS name, not 127.0.0.1, so the generated WebSocket URL points at a host the headset can reach.

  • Optional dashboard query params (only if you use them): ?token=… for hub auth, or ?wsUrl=… if the dashboard’s origin differs from the hub WebSocket host.

2. Copy the pre-built query string from the dashboard

  • In the dashboard, expand Headset.

  • Hub WebSocket shows the raw controlWsUrl value (the wss://…/teleop/v1/ws string).

  • Add to WebXR app URL shows a template: https://<your-teleop-web-client-url>/?controlWsUrl=… (and controlToken=… when a token is in use). The query part is already URL-encoded; keep it intact.

You can either:

  • Replace https://<your-teleop-web-client-url>/ with the real HTTPS base URL where you host the Teleop WebXR client, then copy the entire line and use it as the browser URL on the headset, or

  • Copy only everything from ? onward and append it to your known client base URL (e.g. https://nvidia.github.io/IsaacTeleop/client/?controlWsUrl=…).

3. Move the URL from PC to headset without retyping

Use whichever option your headset browser and IT policy allow:

  • Send the full URL to yourself (email, chat, notes) and open the link in the headset browser.

  • ADB (Android-based headsets, dev kits): with USB or wireless debugging enabled, from the PC run adb shell am start -a android.intent.action.VIEW -d '<full-https-url>' so the system browser opens that URL on the device. Use a single pair of quotes around the entire URL (query strings often contain &). Typical for Meta Quest and other Android XR builds when sideloading or iterating without typing in-headset.

4. Continue the normal activation flow

After the page loads with controlWsUrl set, follow User Activation (arm remote control, then connect from the dashboard).

User Activation#

navigator.xr.requestSession() requires a user gesture on most devices. Recommended flow:

  1. User opens the Teleop web app on the headset.

  2. User taps “Arm remote control” — satisfies the browser’s user activation requirement.

  3. Operator sends connect from the dashboard.

After the first session, subsequent connect / disconnect commands work without further headset interaction.

Trust Model#

The hub controls which CloudXR server the headset connects to. Protect accordingly:

  • Use CONTROL_TOKEN on production deployments.

  • Restrict network access to trusted machines (VPN, LAN).

  • TLS is always enforced (same certificates as the CloudXR WSS proxy).

Local testing (hub + WSS without starting the runtime)#

Hub unit tests (no CloudXR runtime, no WSS — only pytest):

cd src/core/cloudxr_tests/python
pip install -e ".[test]"
pytest -q

Hub + operator UI uses the same code path as production: wss.py on the proxy port, with the CloudXR runtime not started:

python -m isaacteleop.cloudxr --wss-only --enable-hub --accept-eula

(Install isaacteleop from your build or pip install -e as usual.) WSS logs go to stderr. Optional: CONTROL_TOKEN env var; TELEOP_CONTROL_HUB=1 can replace --enable-hub. Non-teleop WebSocket paths still try to proxy to the runtime backend, which is not running in this mode—use teleop URLs only, or run the full stack without --wss-only.

Open https://127.0.0.1:48322/teleop/ in a normal desktop browser (Safari, Chrome, Firefox) and accept the self-signed certificate. Do not use the editor’s embedded browser / Simple Browser / preview pane for this URL — some IDEs then show unrelated RPC / “stateful work request” errors.

If the same RPC message appears in Chrome anyway, the teleop server still does not emit it (no gRPC client). Typical causes on a full browser:

  • A Chrome extension (corporate security, VPN, devtools helpers, vendor tooling) intercepting https://127.0.0.1 or TLS. Try Incognito with extensions disabled, or a fresh browser profile, and check DevTools → Console: if the stack trace points at chrome-extension://…, disable that extension.

  • Managed / enterprise Chrome or local security agents that MITM or classify localhost traffic.

Extra slashes in the path (e.g. //teleop/) are normalized by the server.
